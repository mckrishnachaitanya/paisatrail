# PaisaTrail — Project Handover

Personal expense tracker PWA. Re-paste this into Project knowledge at the start of each new session — it's written to let a fresh Claude conversation pick up without re-deriving prior decisions or re-discovering already-fixed bugs.

## 1. Project facts
- Android-installable PWA, hosted on GitHub Pages. Vanilla HTML/CSS/JS only — no framework, no bundler, no build step, no external libraries. INR only.
- **Status: V1 fully shipped. V2 in progress — Recurring Expenses, both quick wins (amount-range filter, warranty-expiry banner), Cash-Flow Forecasting, Budgets, Biometric Unlock, and Income Tracking all shipped. Biometric confirmed working on device; income tracking not yet device-tested.** Real-device testing happens in separate conversations as Krishna uses the app day-to-day; treat it as an ongoing parallel process, not a gate before building more.

## 2. What's built
- **PIN:** setup/unlock, 5-fail lockout (30s), 2-min idle auto-lock, screen-restore on unlock, in-app Change PIN (reuses the unlock state machine — see §4).
- **Home:** Overview tab (monthly total, category donut, recent list, swipe-to-delete, warranty-expiring-soon banner) + Warranties tab (Active/Expired, soonest-expiring first) + Forecast tab + Budgets tab (see below). 4 tabs total now — confirmed on a real device that `.home-tabs` still reads comfortably at this count, no crowding or wrapping.
- **Add/Edit:** category grid, payment chips, date, category-aware note placeholder, attachments (camera/gallery/file, unlimited, tap-to-view), warranty toggle (6/12/24mo or hand-edited), group toggle (existing or new, created inline).
- **Transactions:** date-grouped, live search, Filters sheet (category + payment multi-select + date range + amount range, live-apply, persists across nav), tap-to-edit, delete.
- **Insights:** Week/Month/Year/Custom period switch; Categories/Compare/Payment views (vs. previous period) + a 4th Groups view (not calendar-bound).
- **Settings:** Categories manager (icon picker expanded to 71 icons across themed groups — food/transport/home/health/fitness/entertainment/finance/travel/family/shopping/misc, `CATEGORY_ICON_CHOICES`; Health deliberately kept to just 🏥+💊 per Krishna's call, not the fuller set originally proposed), Payment Methods manager, Groups manager (rename/delete/merge), Tag Past Expenses (retroactive bulk-tagging by date-range or manual pick), Export/Import (JSON, attachments as base64, import = full replace), Change PIN, bulk-delete-old-expenses, Recurring Expenses manager (add/edit/pause/delete; frequency+start date locked after creation).
- **Recurring expenses:** "Make this recurring" toggle on Add/Edit (Weekly/Monthly/Yearly; shows a locked, read-only info row instead if the expense was itself generated by a rule) plus a dedicated Settings manager. A catch-up pass runs on every unlock, backfilling anything overdue and showing one batched toast ("Netflix added automatically" for one occurrence, "Netflix and Rent added automatically" for exactly two *different* rules, a plain count otherwise). Editing a rule's amount/category/payment/end-date only affects future-generated occurrences — past expenses stay untouched historical records.
- **Cash-Flow Forecasting:** a Home tab with a toggle between "Rest of this month" and "Next 30 days." Projects a total from three parts shown separately for transparency: actual spend so far (month view only), known recurring expenses still due (computed by simulating each rule forward, no DB writes), and an estimate for everything else from a trailing daily average — shown as "Not enough history yet" rather than a fake number when there's no data. The lookback window for that average is user-selectable (30/90/180 days, chips right on the tab). Also lists the specific upcoming recurring items with their due dates.
- **Budgets:** a 4th Home tab (Krishna's call, same reasoning as Forecast). Monthly limits, either per-category or one "Overall" total — managed entirely from this tab (no separate Settings screen), same list-plus-edit-sheet shape as Recurring. Each budget shows as a progress-bar card: spent vs. limit, color shifting mint → sunshine (≥80%) → coral (≥100%), bar visually caps at 100% width even when over budget while the actual spent figure keeps climbing. Overall is pinned first if set; category budgets below it are sorted by how close each is to its limit. "+ Add budget" only offers categories (and Overall) that don't already have one, and hides entirely once nothing's left to add. Editing an existing budget locks which category it's for — only the limit and deleting are available; to redirect a budget to a different category, delete and recreate it.
- Soft-delete-with-undo everywhere; all expense-derived views refresh together after any delete/undo.

- **Income tracking:** Add screen now has an Expense/Income toggle at the top. Income entries use a hardcoded set of 7 categories (Salary, Freelance, Business, Rental, Gift, Investment, Other — stored in `INCOME_CATEGORIES` constant, not user-editable yet) stored alongside expenses in the same IndexedDB `expenses` store with `type:'income'` and `incomeCategoryId` fields. Existing records with no `type` field are treated as expenses via `isExpenseEntry(e)` / `isIncomeEntry(e)` helpers. Home overview card replaced with a 3-stat cashflow layout (Income / Expenses / Net). Transactions list shows income entries in mint green with a + prefix. Filter sheet has a new TYPE toggle (All / Expenses / Income) at the top; category filter hides when Income-only is selected since income has its own fixed categories. Expense-only features (warranty, group, recurring, attachments) are hidden from the Add screen in income mode. All budget, forecast, warranty, insights, and group queries exclude income via `isExpenseEntry` guard. ("Biometric unlock," off by default, `pt_bio`/`pt_bio_cred` in `Store`). Replaces the originally-planned "biometric step-up" concept entirely — Krishna's call after comparing to the password-manager app this was built from: step-up made sense there because each secret inside the vault was independently high-value, but expense data has no equivalent graduated sensitivity once past the PIN, so there's no case for a mid-session re-auth gate. This is purely a faster *unlock* path layered on top of the PIN, never a replacement — PIN remains the underlying credential and the automatic fallback. Turning the toggle on immediately fires the device's WebAuthn platform-authenticator registration prompt (fingerprint/face) via `registerBiometric()`; that prompt succeeding is itself the security checkpoint, no separate PIN re-confirmation needed. When enabled, every arrival at the unlock screen (cold start via `init()`, idle/visibility lock via `lockVault()`) auto-fires `attemptBiometric()` through a shared `enterUnlockScreen()` entry point — Krishna's call, no manual tap required — with a small "Use biometric instead" retry button on the PIN screen as a manual fallback if the OS prompt gets dismissed.

- **Income tracking:** opt-in alongside expenses on the same Add screen via an Expense/Income segmented toggle. Income entries are stored in the same `expenses` IndexedDB store with `type:'income'` and `incomeCategoryId` (pointing into the hardcoded `INCOME_CATEGORIES` list — 7 fixed categories: Salary, Freelance, Business, Rental, Gift, Investment, Other). Expense records written from now on carry `type:'expense'`; records without a `type` field (all existing data) are treated as expenses via `isExpenseEntry()`/`isIncomeEntry()` backward-compat helpers — these helpers are used in every DB query so income never pollutes budgets, forecast, warranty, insights, or bulk-delete. Home screen now shows a 3-stat cashflow card (Income / Expenses / Net) instead of the old single spend total; the recent list shows both types. Transactions list shows income rows in mint with a + prefix; the filter sheet has a new TYPE toggle (All / Expenses / Income) that hides the category filter when Income-only is selected. Expense-only fields (warranty, group, recurring, attachments) are hidden automatically on the Add screen when income mode is active.

## 3. Data model
- **Expenses:** id, amount, categoryId, paymentMethod (stores the *id*, despite the name), date, note, attachmentIds[], recurringId (nullable — set when an expense was generated by, or itself created, a recurring rule), groupId (nullable), warrantyExpiry (nullable — presence = "under warranty"), warrantyDurationMonths (nullable), createdAt.
- **Categories:** id, name, icon, color, isDefault. **Payment methods:** id, name, icon (no color). **Groups:** id, name only.
- **Recurring:** id, name, amount, categoryId, paymentMethod, frequency ('weekly'|'monthly'|'yearly' — no daily), startDate, endDate (nullable), nextDueISO, active, createdAt.
- **Budgets:** id, categoryId (nullable — null means the special "Overall" budget), monthlyLimit, createdAt. At most one budget per categoryId, and at most one with categoryId=null.
- **Attachments:** id, expenseId, type, fileName, mimeType, data (ArrayBuffer).

## 4. Architecture & gotchas to know before touching code
- IndexedDB, no Dexie. `DB_VERSION=4`. Stores: expenses (indexed on date/categoryId/paymentMethod), categories, paymentMethods, groups, recurring, budgets, attachments.
- **No index on `expenses.groupId` or `expenses.recurringId`** — "every expense linked to X" always means pulling the full date-range cursor and filtering in JS (`getExpensesByGroupId`, `getExpensesByRecurringId`).
- Recurring date math always re-derives the target day fresh from the rule's `startDate`, never from the previous occurrence's (possibly already-clamped) date — a rule due "on the 31st" that lands on Feb 28 must come back to Mar 31 next, not drift to Mar 28 forever (`advanceRecurringDate`/`clampToValidDate`). Because of this, frequency and start date are locked read-only once a rule exists (visible in the edit sheet's read-only cadence line) — to change cadence, end the rule and create a new one.
- Catch-up (`runRecurringCatchUpAll`) runs on every unlock, not just cold launch — `unlockSuccess` fires on every PIN entry including resuming from idle auto-lock — capped at `RECURRING_CATCHUP_CAP=24` backfilled occurrences per rule per pass. Toast batching: names the rule for exactly one occurrence, names two only if they're genuinely two *different* rules ("Netflix and Rent"), otherwise falls back to a plain count — never "Rent, Rent, and Rent."
- A rule created via the Add/Edit toggle sets `nextDueISO` to the occurrence *after* the expense being saved (that expense itself is occurrence #1, written directly, not by catch-up). A rule created via Settings' "+ Add" has no first occurrence yet, so it sets `nextDueISO` to the chosen start date and runs the single-rule catch-up function once immediately to backfill if that date is today or earlier.
- No data encryption — PIN + idle-lock only, with biometric unlock now layered on top as a faster front door (see §2) — still no encryption at rest either way.
- The Home warranty-soon banner reuses the existing warranty data — no new store. Window is `WARRANTY_SOON_DAYS=30`, already-expired items are deliberately excluded (they belong to the Warranties tab's Expired group, not this banner). Names the item if exactly one is due, falls back to a count otherwise — same naming convention as the recurring catch-up toast.
- Transactions' amount-range filter follows the date-range filter's exact shape: independent min/max bounds, either can be set alone, and the pair counts as a single filter "category" for the Filter button's badge — mirrors how FROM/TO already work, don't treat them as two separate filters.
- Forecasting (`projectRecurringTotal`) is a pure read-only simulation — it reuses `advanceRecurringDate` to walk a *local* cursor variable forward, never touching `rule.nextDueISO` or writing to IndexedDB, so it's safe to call on every render. The "estimated other spending" average (`estimateDailyOtherSpend`) explicitly excludes expenses with a `recurringId` — otherwise recurring spend would get counted twice, once as a known line item and again baked into the variable-spend average. No new DB store was needed for this feature.
- A rule just created from the Add/Edit toggle won't show up in "Rest of this month" the same day — its `nextDueISO` is already advanced to the *next* period (today's own expense is occurrence #1), so a same-day-created monthly rule's next occurrence is next month by definition. This is correct, not a bug; it shows up once the window widens (e.g. the rolling 30-day view) or once the next period actually arrives.
- Forecasting has two genuinely different "30 days" that must stay decoupled: `FORECAST_ROLLING_WINDOW_DAYS` (fixed at 30, defines the forward "Next 30 days" view itself) and `state.forecastLookbackDays` (user-configurable 30/90/180, how far *back* the "estimated other spending" average looks). They started as the same constant before the lookback became configurable — don't let them merge back into one variable, or changing the lookback chip would silently also change what "Next 30 days" means.
- `estimateDailyOtherSpend` requires both `FORECAST_MIN_EXPENSE_COUNT=5` expenses *and* `FORECAST_MIN_DISTINCT_DAYS=7` distinct days before trusting the average — not just "any data at all." Without the distinct-days check, a single large one-off purchase (or several expenses logged on the same day) would get projected forward as if it were routine daily spending. Found via a real walkthrough: a single ₹25,500 expense on a fresh app was inflating the projected total by another ₹10,000+ of phantom "estimated other spending" built from that one purchase.
- Budgets' spent-so-far calculation deliberately *includes* recurring-generated expenses (unlike Forecast's estimate, which excludes them) — a recurring Rent charge in the Housing category is genuinely real spend in that category, no double-counting risk the way Forecast had. `budgetSpentAmount`/`getMonthToDateExpenses` are budgets-only helpers, separate from anything in the Forecasting section.
- A budget's `categoryId` is locked once created — only `monthlyLimit` is editable afterward (mirrors Recurring's frequency/start-date lock for the same reason: redirecting it would be confusing rather than useful; delete and recreate instead). Deleting the category a budget points to also deletes that budget (see `confirmManageDelete`'s category branch) — same "drop now-meaningless dependents" logic already used for Transactions filters.
- Export/Import: single JSON, attachments base64-encoded (chunked at 0x8000 bytes — `String.fromCharCode.apply`'s argument limit). PIN is deliberately excluded from exports. Import is a full **replace** (one atomic multi-store transaction, now including `recurring` and `budgets`), rejecting files missing the PaisaTrail marker or with empty categories/paymentMethods arrays. `schemaVersion` is 3 now (bumped again when budgets were added to the export, having already been bumped once for recurring rules) — old backups from either earlier schema still import fine, just restoring nothing for whichever store(s) they predate.
- Deleting a category/payment-method/group **unlinks**, never cascades — except the very last remaining category or payment method, which is blocked outright. Confirm dialogs show a usage count.
- Groups are created lazily (only on actual Save/confirm) from Add/Edit or Tag Past Expenses — never write a group to IndexedDB until the action that uses it is confirmed, to avoid orphans.
- **`esc()` is for `innerHTML` only.** Never apply it before a `.textContent` assignment — it double-escapes (e.g. shows `&amp;` literally). Caused a real bug once; fixed, but watch for it in new code.
- Several confirm-and-close functions close their overlay **before** their awaited DB write finishes. Don't treat "overlay closed," or even matching toast text (messages can repeat verbatim across consecutive actions), as proof the DB/state actually updated.
- Change PIN reuses the pin-screen state machine (`change-verify`/`change-new`/`change-confirm` modes) and shares `verifyAgainstStoredPIN` (hash-check + 5-strikes lockout) with normal unlock.
- View-state (search, filters, Insights period/view, `state.homeTab`) persists across navigation by convention — don't reset it on revisit unless explicitly asked.
- Two near-identical segmented-control implementations exist (`.home-tabs`, `.segmented-tabs`) — reuse one of these in new code, don't add a third.
- **`renderTransactions()` is now the single source for both the visible list and the filter sheet's "Show N expenses" button** — one `getFilteredTransactions()` call feeds both, guarded by a module-level `txnRenderSeq` counter. Each call captures its own sequence number and bails before touching the DOM if a newer call has started while it was awaiting the DB — without this, rapid filter taps could let an older, slower query resolve after a newer one and briefly leave the button showing a stale count (real bug Krishna caught on-device; fixed this session). Any new code that reads filtered-transaction state asynchronously and writes to the DOM should follow the same guard pattern rather than firing an unguarded parallel query.
- **Biometric unlock feasibility (resolved):** WebAuthn's platform authenticator works identically inside an Android "Add to Home Screen" PWA (WebAPK) as in a regular Chrome tab — installed/standalone display mode doesn't gate the API, only secure context (GitHub Pages HTTPS, already satisfied) and the platform's own authenticator support (mature for years on Android Chrome). No native wrapper or Trusted Web Activity / Digital Asset Links setup needed — that's a different, heavier approach this app isn't using. One real risk worth remembering if this gets revisited: the registered credential ID lives in `localStorage` (`pt_bio_cred`), but the actual key lives in Android's secure hardware — if Chrome's site data ever gets cleared or the PWA is reinstalled, the stored ID is lost even though the platform-side credential may still exist. `attemptBiometric()`'s try/catch already treats any failure (including a now-missing/invalid credential) as a silent fall-through to PIN, so this degrades gracefully without extra handling.
- `enterUnlockScreen()` is the single entry point for showing the PIN screen in unlock mode (used by both cold-start `init()` and `lockVault()`) specifically so the biometric auto-attempt fires exactly once per lock — don't call `initPINScreen('unlock')` + `showScreen('pin-screen')` directly from new code, route through this instead, or biometric won't auto-fire there.

## 5. Testing
- Playwright + Chromium is available in the build sandbox — use it for real headless-browser smoke tests on every feature built, not just static syntax/ID checks. It has caught real bugs static analysis missed.
- **Scope every nav/FAB selector**: bottom-nav and the FAB are duplicated once per main screen (only one copy is visible at a time). A bare `[data-nav="x"]` can silently click a hidden screen's copy — always use `.screen:not(.is-hidden) [data-nav="x"]`.
- Poll the actual resulting DOM/state directly after an action, not an intermediate signal (see overlay-close gotcha above).
- When testing the catch-up engine in isolation, deactivate any rules created earlier in the same test run first — `runRecurringCatchUpAll` walks *every* active rule, so an earlier rule that's quietly become overdue (nothing has caught it up yet) will fire alongside whatever you're specifically testing and throw off occurrence counts.
- When testing forecasts: remember a same-day-created rule's `nextDueISO` is already next period (see the gotcha above) — don't write a test asserting it appears in "Rest of this month" the same day it's created; assert it appears once the window widens instead.
- When testing Budgets: the budget-picker grid and the Add/Edit screen's category grid are NOT in the same order — the budget picker has "Overall" prepended as chip #1, so the same category sits one slot later there than it does in Add/Edit's grid. Picking "chip #2" in one doesn't mean the same category as "chip #2" in the other; index off by one here was a real test bug caught during this build, not an app bug.
